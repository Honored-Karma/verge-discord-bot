Система слотов персонажей

Кратко

- Один пользователь может иметь несколько персонажей (слотов): слот 1, слот 2, ...
- У пользователя всегда есть активный слот — команды работают с активным персонажем.
- Слоты хранятся в MongoDB коллекции `players` как отдельные документы с `id = "<userId>_<slot>"` и полями `user_id` и `slot`.
- Текущий активный слот хранится в коллекции `active_slots`:
  { user_id: "<userId>", slot: <number> }

Структура данных (пример)

players (collection):
{
  id: "123456789012345678_1",
  user_id: "123456789012345678",
  slot: 1,
  username: "DiscordName",
  character_name: "Akira",
  character_avatar: "https://...",
  krw: 0,
  yen: 0,
  ap: 0,
  ap_multiplier: 100.0,
  sp_multiplier: 100.0,
  // ...другие поля
}

active_slots (collection):
{
  user_id: "123456789012345678",
  slot: 1
}

inventory (collection):
{
  player_id: "123456789012345678_1", // соответствует players.id
  item_name: "+Техника",
  qty: 2
}

player_sp (collection):
{
  player_id: "123456789012345678_1",
  style_id: 5,
  sp: 120
}

Основные функции/поведение

- getPlayer(identifier)
  - Если передан точный `id` (например `"123_1"` или устаревший `"123"`), возвращает соответствующий документ.
  - Если передан `userId` (например `"123"`), резолвит активный слот в `active_slots` (по умолчанию 1) и возвращает документ `players` для `user_id + slot`.
  - Это позволяет существующим командам не менять код: они продолжают вызывать `getPlayer(userId)` и получать активного персонажа.

- createPlayer(userId, username, characterName, avatarUrl, slot = 1)
  - Создаёт документ в `players` с `id = `${userId}_${slot}``.
  - Если `active_slots` для пользователя ещё не задан, устанавливает созданный слот как активный.

- setActiveSlot(userId, slot)
  - Обновляет/создаёт запись в `active_slots`.

Команды (реализация)

- /slot <number>
  - Переключает активный слот для пользователя.
  - Если в указанном слоте нет персонажа — слот будет установлен, но бот скажет, что слот пуст; для создания персонажа используйте `/register slot:<n>`.

- /register character_name:<name> [avatar:<file>] [slot:<n>]
  - Если параметр `slot` указан — регистрирует персонажа прямо в указанном слоте (если он свободен) и делает его активным.
  - Если параметр `slot` не указан — создаёт персонажа в текущем активном слоте (по умолчанию 1).

Миграция и совместимость

- Старые записи, где `players.id === <userId>`, по-прежнему поддерживаются — `getPlayer` сначала пытается найти документ с точным `id`.
- Для перехода на много-слотовую модель можно: для каждого существующего пользователя скопировать/переименовать документ в `id = "<userId>_1"` и установить `user_id = <userId>, slot = 1`.

Примеры

- Получение активного персонажа (в коде):
  const player = await getPlayer(userId);

- Создание персонажа в слоте 2:
  await createPlayer(userId, username, 'Ken', avatarUrl, 2);

- Переключение слота:
  await setActiveSlot(userId, 2);

Замечания

- Inventory и SP-таблицы теперь хранят `player_id` как ссылку на конкретный слот (например `"123_2"`).
- Большая часть старого кода остаётся без изменений — достаточно, что `getPlayer` возвращает нужный документ.
